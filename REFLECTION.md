# REFLECTION

本项目的目标是构建一个 Safe Test-Repair Coding Harness，也就是面向代码修复场景的安全型智能体运行框架。相比直接依赖大模型自由发挥，我选择把重点放在 harness 本身的工程机制上：由代码实现主循环、动作解析、工具分发、测试反馈、记忆记录、停止策略和安全护栏，让智能体能够在受控边界内读取文件、运行测试、根据失败反馈调整下一步行动，并最终完成 mock 驱动的修复流程。这个项目不是一个聊天机器人，而是一个 coding agent harness。它的价值不在于和用户自由对话，而在于展示一个可验证的代码修复闭环：先运行测试，得到客观失败信息，再改变下一步 action，最后重新测试并停止。
项目的核心机制是自己实现的 agent loop。一次修复过程会被拆成多个明确步骤：mock LLM provider 先生成下一步动作，action parser 把模型输出解析成结构化 action，guardrail 检查动作是否安全，例如是否越界写文件、执行危险命令或访问敏感路径。只有通过检查后，tool dispatcher 才会调用受控工具，比如读写文件或运行测试。测试输出会交给 feedback analyzer，它会把 pytest 结果分类为测试通过、断言失败、导入错误、语法错误、超时等稳定反馈，再进入下一轮上下文。memory 使用 JSONL 记录每轮事件，stop policy 根据测试是否通过、最大轮数和错误状态决定是否停止。我的主要贡献就是 deterministic feedback loop 和 code-level guardrail：前者让 agent 不靠自我声明“修好了”，而是靠 pytest 反馈推进；后者让安全边界不依赖 prompt，而是由代码强制执行。
开发过程中，我觉得 Superpowers 里最有用的是 TDD、planning 和 review 相关流程。它们逼着我先写 SPEC / PLAN，再按任务拆分实现，而不是一边写一边改方向。最“形式大于实质”的部分是有些状态更新和文档记录比较机械，比如每个 task 都要更新 PLAN、AGENT_LOG、commit hash、review outcome，过程比较繁琐。但它也不是完全没意义，因为到最后做 Task 19/20 acceptance sweep 时，这些记录确实帮我证明每一步是怎么完成的。总体来说，TDD 在 AI 协作下不是阻碍，而是放大器。单看某一步会觉得慢，因为要先 red 再 green；但对 AI 来说，测试就是边界。没有测试时，agent 很容易写出看起来合理但范围过大的代码；有测试时，它更像是在一个小框里完成任务。
subagent-driven 工作流能让智能体自主运行多久，取决于任务颗粒度。我没有严格计时，但从体验上看，一个 2-5 分钟、只改一个模块和对应测试的任务最稳。比如只做 feedback analyzer、只做 guardrail、只做 CLI demo，这类任务 agent 可以比较久不偏离主题。反过来，如果一个 prompt 同时要求实现 agent loop、WebUI、Docker 和 README，就很容易混在一起，甚至提前实现后续任务。所以我认为最优颗粒度是：目标清楚、文件范围有限、测试能独立验证，并且完成后能形成一个可 review 的机制切片。
SPEC / PLAN 的质量直接影响实现质量。一个具体例子是 CI 任务最开始只写了要有 GitHub Actions 跑测试，但没有把“干净环境里需要显式安装 pytest”说得足够具体，结果实现后 review 发现 CI 可能无法运行 pytest，后来补了回归检查和配置。还有 tool dispatcher 的 `run_tests` observation 一开始保存了完整 stdout/stderr，这说明如果规约只说“记录测试输出”，但没有强调 bounded excerpt，subagent 就会倾向于把所有内容都塞进去，造成 memory 或 WebUI trace 过长。后来通过 review 和回归测试把它改成只保留 bounded excerpt。这让我意识到，SPEC 不能只写功能，还要写边界、负例和不能做什么。
我最有效的 prompt/context 策略是每次只给 Codex 一个 task，并明确写清楚：只修改哪些文件、不能开始下一个 task、必须先 red 再 green、运行哪些测试、最后不要提交。这样做有效，是因为 AI 很容易“顺手多做”，而这些限制能把它关在任务范围里。另一个有效策略是把 review 也当成独立任务，让它只检查不修改。这样实现和审查分开，比较容易发现越界、遗漏和安全问题。
凭据与分发这两条工程要求也迫使我想清楚了一些原本会忽略的问题。凭据方面，我不能把真实 API key 写进代码、测试、CI、Docker 或 README，也不能让测试依赖本机 keyring，所以项目采用 mock/stub provider 和 fake secret store。分发方面，Docker、GitHub Actions、GitLab CI 和 Render 部署要求我考虑“别人拿到项目后能不能跑”，而不只是我本机能跑。比如 Dockerfile 只复制 pyproject 和 src，默认运行 mock demo；GitHub Actions 在 push 时跑 pytest 并构建 Docker；WebUI 部署后还要验证 `/health`、`/demos/repair-loop`、`/upload` 和 workspace query 的行为。
如果重做，我会更早把最终 README contract 和部署 contract 测试写出来，这样后期 Task 18/19 不会集中补很多文档要求。我也会更早决定 WebUI 只是 JSON demo endpoint，而不是等到后面再解释它为什么不是聊天界面或完整前端。对于 Open Design 这类 UI 要求，我也会在 SPEC 里提前说明：本项目主要是 CLI / 后端式 mock endpoint，没有做自定义前端界面，因此按纯 CLI / 后端展示豁免。
最后，我对 Superpowers 方法论的批判是：它假设任务可以被清楚拆分，测试能表达需求，review 能及时发现偏差，开发者也愿意维护大量过程文档。在我的项目里，这些假设大部分成立，因为 harness 本身适合模块化，parser、guardrail、feedback、memory、stop policy 都能写成独立测试。但它也有成本：文档和流程负担很重，有时会让人感觉在服务流程而不是服务功能。我的体会是，这套方法适合安全性、可审计性、可复现性要求高的 agent 项目；但如果是探索性很强、需求还没稳定的项目，它可能会显得过重。对我来说，这次最大的收获不是写出了一个多强的 agent，而是理解了 agent 外层工程系统的重要性：真正可靠的 coding agent 不应该只相信模型，而应该依靠测试反馈、安全边界和可追溯流程来约束模型。
